# 0016 Enterprise Lineage Boundary

Date: 2026-06-16

## Status

Accepted

## Context

VSF Data Profiler now produces schema parse diagnostics, connector metadata,
schema evaluation, relationship graph, dataset verdict, chart specs, runtime
events, and reports. Users need a single dependency graph that explains where
data came from and which tables, columns, stages, and artifacts support each
result. This changes the public artifact set and touches connector-derived
metadata, so the boundary needs a durable decision.

## Decision

Add `lineage_graph.json` as an additive local artifact generated by the
existing Python pipeline after deterministic machine artifacts are available.
The graph is evidence-only and may reference:

- CSV or connector source metadata;
- DBML or connector-introspected schema entities;
- relationship graph edges;
- runtime stages and events;
- generated artifact paths and known source-artifact dependencies.

Lineage must not read raw CSV rows, publish metadata externally, write to user
databases, or infer unsupported transform lineage. Connector-derived values use
the same redaction rules as runtime and connector metadata before they enter
the lineage artifact.

Reports and the local web runner expose `lineage_graph.json` like other
canonical artifacts. The web dashboard may link or display lineage metadata
from generated artifact URLs, but remains an artifact consumer and not a
profiling or lineage inference engine.

## Alternatives Considered

1. Publish lineage to an external catalog. Rejected because the product remains
   local-first and external metadata publishing is out of scope.
2. Add transform-level SQL lineage. Rejected because current artifacts prove
   profiling dependencies, not arbitrary transformation semantics.
3. Build lineage only in JavaScript. Rejected because machine lineage should be
   reproducible from the pipeline and available to CLI users.

## Consequences

Positive:

- CLI and web users get a single machine-readable lineage inventory.
- Existing artifacts remain unchanged and become lineage evidence.
- Connector source lineage stays compatible with the established redaction
  boundary.

Tradeoffs:

- The graph is intentionally coarse for transformations and does not claim
  dbt/Airflow/BI lineage.
- Lineage reflects artifacts available at report-render time; final runtime
  summary still remains the authoritative terminal run status.

## Follow-Up

- Consider richer transform lineage only if future artifacts prove the needed
  semantics without raw-row reads or external catalog coupling.
