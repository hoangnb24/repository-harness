# US-106 V1 Phase 1 Exec Plan

Status: **Implemented / accepted**

## Goal

Implement and prove every Phase 1 acceptance item in the refactor plan and
US-105, then make Phase 2 ready without implementing Phase 2 core behavior or
Phase 4 bridge conversion.

## Scope

In scope:

- Accepted Decision 0013 and this high-risk packet.
- Versioned prose/JSON contracts, complete path/data dispositions, exact V0
  schemas and parser/feature snapshots.
- Deterministic positive/negative fixtures and focused enforcement.
- Premerge integration through the repository's existing validation script.
- Truthful Phase 1 evidence/status updates after all proof passes.

Out of scope:

- V1/bridge binaries, runtime behavior, installer mutation, production trust
  material, signing/publishing, release workflows, pilots, and Phases 2–8.
- V0 behavior changes or root Harness state operations.
- Push, tag, release, deploy, PR, external repository, or Herdr mutation.

## Risk Classification

Risk flags:

- Authentication, authorization, bootstrap, rollback, and revocation.
- Data loss through SQLite WAL capture or archive tamper.
- Public CLI, schema, release, path, and compatibility contracts.
- Cross-platform path identity, case, Unicode, symlink/reparse behavior.
- Existing V0 behavior freeze and payload separation.

Hard gates:

- G0 is accepted by Decision 0012.
- Security/data-integrity choices are accepted by Decision 0013.
- No Phase 1 completion claim until focused, affected, formatting/lint,
  whitespace, and full premerge proof all pass.

## Work Phases

1. Inspect the current manifest, installers, release workflow, schemas, Rust
   parser, public command manifest, templates, and US-105 acceptance criteria.
2. Record Decision 0013 and create the US-106 packet from high-risk templates.
3. Define versioned manifest, command, trust, audit/scaffold, compatibility,
   capture, archive, availability, and retirement contracts.
4. Inventory every current install/release path and V0 durable/filesystem
   category with one disposition.
5. Freeze exact schema bytes, command/capability identity, and changeset matrix.
6. Add deterministic test-only signatures, path/grammar/schema negatives,
   archive tamper, and real DB/WAL/SHM fixtures.
7. Add one focused verifier and premerge integration; prove its negatives.
8. Run focused and full validation, inspect scope/generated bytes, then update
   Phase 1/US-105 evidence truthfully.
9. Use the smart-commit workflow to stage only inspected Phase 1 files and
   create one conventional commit without pushing.

## Stop Conditions

Stop and request human direction if:

- a required contract conflicts with Decision 0012 or the approved security
  package;
- exact current V0 behavior cannot be frozen without modifying it;
- proof would require a production key, release/publish action, Phase 2 binary,
  or Phase 4 conversion write;
- an existing unrelated/user change overlaps the owned paths; or
- verification would need to weaken a fail-closed rule.
