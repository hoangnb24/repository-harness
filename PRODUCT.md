# Product

## Register

product

## Users

Non-technical task owners use Symphony Web UI to start, watch, review, and approve local Harness Symphony work without operating the CLI. Technical maintainers use it to inspect run evidence, logs, PR state, sync state, and validation proof from one local controller.

## Product Purpose

Symphony Web UI is a local, unauthenticated controller for existing Harness and Symphony concepts. Success means users can understand work state, make safe control decisions, and keep durable Harness records as the source of truth.

## Brand Personality

Calm, operational, trustworthy. The interface should feel like a focused command center for repeated work, with clear state and restrained polish rather than marketing drama.

## Anti-references

Avoid landing-page composition, decorative dashboards, invented controls, hidden unsafe actions, and UI that creates a second source of truth outside Harness. Avoid decorative glass, large rounded cards, gradient text, and heavy color on inactive states.

## Design Principles

- Put the work board and current state first.
- Use familiar product controls and local shadcn-style primitives.
- Keep irreversible or workflow-changing actions explicit and guarded.
- Prefer scanability, stable density, and readable evidence over ornament.
- Make backend writes and Symphony execution separate, visible choices.

## Accessibility & Inclusion

Target accessible product UI basics: keyboard-reachable controls, visible focus, semantic loading/error/status states, reduced-motion alternatives, bounded text, and contrast suitable for repeated operational use.
